Our third assignment for “Interactive Media: Web” was to create a web page that encapsulated some dimension of something. The only the requirements were that the web page was to either include a Javascript library, or was pull from an API of some sort. I had a lot of different ideas for various web pages (such as Pirate Radio Station, a Last.FM remake just for myself, a game of some sort), but ultimately I ended up settling on creating an audio/visual demonstration. This web page is an encapsulation of a curated system of sounds.

Shaheer mentioned the reasoning for having such an open ended assignment was to allow us to do whatever we want with it. I have recently fell into the habit of buying music making software from random online sales and while I have dozens of new applications ready to play around with, I just have not found the time to actually try them out. I guess you could say the main inspiration for this project is the fact that it’s almost vacation time and I really just want the projects to end so I can mess around with music stuff again.

The making of music through code in general has always been something that has interested me. Here some fun examples that I always like to look back on:

  • Pure Data Funky by mrp805
  • Aphex Flim Generator (in SuperCollider) by rukano
  • Kindohm Live @ ICLC 2016, Hamilton, Ontario
  • 11EDO algorithmic IDM in Pure Data by acriel
  • Random PC-DOS Demoscene Mix

The actual making of this assignment was a pain in the ass. This is because Web Audio is not great for real-time music synthesis. Or maybe it is, and I just can’t figure out how to make it great at this given time. Either way, real-time signal synthesis is very taxing process and web browsers are not made to deal with it. This makes total sense if one were to think about it. A digital audio signal is a simulation of “real” audio signal. In order to achieve that sense of “realness” the CPU must calculate about 44100 samples per second (or more of the user demands it). That is over 2.5 million calculations per minute. I believe if I were to do this project on a bigger scale, I would want to encorporate server-side scripting to handle audio processing for me.

I once again used ToneJS to produce most of the sounds for this assignment. The drums however are pre-rendered samples I obtained from a Samples from Mars sample pack. At some point, I was thinking of switching to WebPD for the audio as I do have knowledge of Max/MSP and Pure Data. However, after trying it out I realized that the library came with its own bag of issues. The visuals were created using the P5JS library. I was initially planning to simply create a CSS grid which would change in properties according triggers within the song, but I soon realized ToneJS’s transport function does not like calling DOM changes due to the AudioContext running on a different thread than the document. I was initially worried having P5JS on top of ToneJS would cause more performance issues but surprisingly, I was wrong.
